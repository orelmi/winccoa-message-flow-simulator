# CLAUDE.md — WinCC OA Message Flow Simulator

## Project context

This is a **standalone public repository** containing an interactive HTML simulator that visualizes message flows between WinCC OA managers (dpConnect, dpSet, dpGet, and advanced integration scenarios).

**Audience**: WinCC OA trainers worldwide. The tool is shared across the global Siemens WinCC OA trainer community as a teaching aid. All UI and documentation should be in English; step descriptions are bilingual FR/EN.

## Build rules

### Never edit the generated file
Always edit files in `src/`. After any modification, rebuild:
```bash
python build.py
```
The build script inlines CSS, JS, and logo (base64) into a single self-contained HTML and copies it to `docs/index.html` for GitHub Pages.

### Backup before regeneration
Before generating a new version, **save the current version** in `backup/`:
- Format: `backup/winccoa-message-flow_YYYYMMDD_HHMM.html`

## Adding a new scenario

Create a file in `src/scenarios/` (e.g., `myscenario.js`). It must:
1. Add its i18n step keys: `Object.assign(i18n.fr, {...}); Object.assign(i18n.en, {...});`
2. Define its async scenario function using core APIs (`waitForStep`, `showStep`, `addMessage`, etc.)
3. Self-register: `registerScenario('myScenario', scenarioMyScenario);`
4. If extra managers are needed, add to `scenarioVisibility`

No changes to core files needed. The build script auto-discovers `src/scenarios/*.js`.

## Language rules

### Interface and documentation
- UI labels, buttons, legends, panel titles: **English only**
- Step descriptions and scenario introductions: **bilingual FR/EN**
- A FR/EN toggle switches only step descriptions, not UI chrome
- Language is auto-detected from `navigator.language`; defaults to English
- README, commit messages, PR descriptions: **English**

### No emojis
Do not use emojis in code, comments, or generated content.

## Manager visibility

- Base managers (PLC, Driver, EV, DM, CTRL PID, 2x UI, NGA, PostgreSQL) are always visible
- External managers (Kafka, MCP, MQTT/UNS, Claude Desktop App, etc.) appear only when their scenario is selected
- "Inside WinCC OA" managers (JS Manager, MCP Server, MQTT Publisher) are positioned inside the WinCC OA frame
- "Outside WinCC OA" managers (Kafka, Claude Desktop App, Mosquitto, MES, Cloud Historian) are positioned outside the frame
- Visibility is defined in `scenarioVisibility` and applied via the `visibleManagers` set

## Specification coherence

The generated HTML must faithfully reflect:
- Scenarios and steps defined in `documentation/specification/04-simulator-spec.md`
- Manager names, DPE names, and callback names
- Message ordering from sequence diagrams
- Key rules from `documentation/specification/08-key-rules.md` (everything routes through the EV, etc.)

## WinCC OA conventions (for scenario content)

### Datapoint naming
- Thermocouples: `TC_101` (control), `TC_102` (charge 1), `TC_103` (charge 2)
- Heating elements: `HR_TOP`, `HR_BOT`, `HR_LEFT`, `HR_RIGHT`
- PID regulators: `PID_TOP`, `PID_BOT`, `PID_LEFT`, `PID_RIGHT`

### Technical accuracy
- Configuration DB: **SQLite**
- Archiving: **NGA** with **PostgreSQL** (or SQLite for small installations)
- Never mention RAIMA (obsolete)
- WinCC OA certifications: **IEC 62443-4-1 & 4-2** (ML3), **SIL 3**, NIS2 compliance
